# Claude Bridge -- Data Layer & Session Model

> Detailed specification of the session model, SQLite schema, workspace structure,
> agent file format, configuration, file ownership, data lifecycle, and Auto Memory
> integration.

---

## 1. Session Model

A **session** is the fundamental unit of Claude Bridge. It represents the pairing of
an **agent role** (e.g., `backend`, `frontend`, `devops`) with a **project directory**
(e.g., `/Users/hieutran/projects/my-api`).

### Session ID Derivation

```
session_id = "{agent_name}--{project_basename}"
```

The double-dash `--` separator is intentional: agent names and project basenames use
single dashes, so `--` is unambiguous and reversible.

| Agent Name | Project Path                     | Session ID        |
|------------|----------------------------------|-------------------|
| backend    | /Users/hieutran/projects/my-api  | backend--my-api   |
| frontend   | /Users/hieutran/projects/my-web  | frontend--my-web  |
| devops     | /Users/hieutran/projects/infra   | devops--infra     |
| backend    | /Users/hieutran/projects/other   | backend--other    |
| frontend   | /Users/hieutran/projects/my-api  | frontend--my-api  |

Key properties:

- **Same agent, different projects.** `backend--my-api` and `backend--other` are
  independent sessions with separate context, memory, and workspaces. The backend
  agent role is reused, but each project gets its own conversation history.
- **Same project, different agents.** `backend--my-api` and `frontend--my-api` both
  operate on the same codebase but have different roles, tool permissions, and
  system prompts. They share the project's Auto Memory but have separate Claude Code
  sessions.
- **Uniqueness constraint.** The combination `(agent_name, project_dir)` is the
  primary key. The derived `session_id` is UNIQUE across all sessions.

### What Each Session Has

Every session maps to exactly four artifacts:

```
Session: backend--my-api
    |
    +-- Agent .md file      ~/.claude/agents/bridge--backend--my-api.md
    |                       Defines role, tools, hooks, model, isolation.
    |                       Generated by Bridge, read by Claude Code.
    |
    +-- Workspace           ~/.claude-bridge/workspaces/backend--my-api/
    |                       Task results, stderr logs, metadata.
    |                       Owned entirely by Bridge.
    |
    +-- Claude Code session --session-id backend--my-api
    |                       Persistent conversation context across tasks.
    |                       Owned by Claude Code. Bridge only passes the flag.
    |
    +-- Auto Memory         ~/.claude/projects/<encoded-path>/memory/
                            Learned patterns, preferences, project knowledge.
                            Owned by Claude Code. Bridge reads but never writes.
```

### Session ID in the CLI

When dispatching a task, the session ID flows through the entire stack:

```
bridge-cli.py dispatch backend "add pagination"
    --> looks up agent_name="backend" in SQLite
    --> gets session_id="backend--my-api", project_dir="/Users/.../my-api"
    --> spawns:
        claude \
          --agent ~/.claude/agents/bridge--backend--my-api.md \
          --session-id backend--my-api \
          -p "add pagination" \
          /Users/hieutran/projects/my-api
```

## 2. SQLite Schema

The database lives at `~/.claude-bridge/bridge.db`. It uses WAL (Write-Ahead Logging)
mode for concurrent reads from the Bridge Bot, CLI, and on-complete hook.

### Full Schema

```sql
-- Enable WAL mode for concurrent access.
-- The Bridge Bot reads while on-complete.py writes, and the watcher reads
-- while dispatching new tasks. WAL prevents readers from blocking writers.
PRAGMA journal_mode = WAL;

-- Foreign keys must be explicitly enabled in SQLite.
PRAGMA foreign_keys = ON;


-- ============================================================
-- AGENTS TABLE
-- ============================================================
-- One row per session (agent + project combination).
-- Primary key is the natural key (name, project_dir) because
-- lookups happen by agent name (from Telegram commands).
-- session_id is derived but stored for fast joins with tasks.

CREATE TABLE agents (
    name          TEXT    NOT NULL,    -- Agent role: "backend", "frontend", "devops"
    project_dir   TEXT    NOT NULL,    -- Absolute path: "/Users/hieutran/projects/my-api"
    session_id    TEXT    NOT NULL UNIQUE,  -- Derived: "backend--my-api"
    agent_file    TEXT    NOT NULL,    -- Path to .md: "~/.claude/agents/bridge--backend--my-api.md"
    purpose       TEXT    NOT NULL,    -- Human description: "REST API development"
    state         TEXT    NOT NULL DEFAULT 'idle',  -- idle | busy | error
    total_tasks   INTEGER NOT NULL DEFAULT 0,       -- Lifetime task count
    created_at    TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    updated_at    TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),

    PRIMARY KEY (name, project_dir)
);

-- Fast lookup by session_id (used by on-complete.py and task queries).
CREATE UNIQUE INDEX idx_agents_session_id ON agents(session_id);


-- ============================================================
-- TASKS TABLE
-- ============================================================
-- One row per dispatched task. Linked to an agent session.
-- Only one task per session can be in 'running' status at a time
-- (enforced by application logic, not a DB constraint, because
-- the single-task model is an MVP simplification).

CREATE TABLE tasks (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id      TEXT    NOT NULL REFERENCES agents(session_id) ON DELETE CASCADE,

    -- Input
    prompt          TEXT    NOT NULL,    -- The user's task prompt

    -- Execution state
    status          TEXT    NOT NULL DEFAULT 'pending',
                    -- pending  : inserted, not yet spawned
                    -- running  : subprocess alive, PID tracked
                    -- done     : completed successfully
                    -- failed   : completed with error (non-zero exit or is_error)
                    -- timeout  : killed by watcher after exceeding time limit
                    -- killed   : killed by user via /kill command
    pid             INTEGER,            -- OS process ID of claude subprocess
    exit_code       INTEGER,            -- Process exit code (0 = success)
    error_message   TEXT,               -- stderr or error description on failure

    -- Results (populated by on-complete.py from Claude JSON output)
    result_file     TEXT,               -- Path: workspaces/{session}/tasks/task-{id}-result.json
    result_summary  TEXT,               -- One-paragraph summary extracted from result
    cost_usd        REAL,               -- API cost in USD (from Claude output metadata)
    duration_ms     INTEGER,            -- Wall-clock duration in milliseconds
    num_turns       INTEGER,            -- Number of agent turns (tool calls)

    -- Reporting
    reported        INTEGER NOT NULL DEFAULT 0,  -- 0 = not yet sent to Telegram, 1 = sent

    -- Timestamps
    created_at      TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    started_at      TEXT,               -- When subprocess was spawned
    completed_at    TEXT                -- When result was recorded
);

-- Find running tasks for a session (dispatch checks "is agent busy?").
CREATE INDEX idx_tasks_session_status ON tasks(session_id, status);

-- Find all running tasks (watcher checks PIDs).
CREATE INDEX idx_tasks_running ON tasks(status) WHERE status = 'running';

-- Find unreported completions (Bridge Bot polls for results to send).
CREATE INDEX idx_tasks_unreported ON tasks(reported, status)
    WHERE reported = 0 AND status IN ('done', 'failed', 'timeout', 'killed');
```

### Query Patterns

**Get agent by name (most common -- Telegram sends agent name only):**

```sql
-- When user says "/task backend add pagination", we need the full session info.
-- In MVP, agent names are unique (one project per agent name).
-- Phase 2 may require disambiguation if same name used for multiple projects.
SELECT session_id, project_dir, agent_file, purpose, state
FROM agents
WHERE name = ?;
```

**Check if agent is busy before dispatching:**

```sql
SELECT id, prompt, started_at
FROM tasks
WHERE session_id = ? AND status = 'running'
LIMIT 1;
```

**Insert new task:**

```sql
INSERT INTO tasks (session_id, prompt, status)
VALUES (?, ?, 'pending')
RETURNING id;
```

**Mark task as running (after subprocess spawns):**

```sql
UPDATE tasks
SET status = 'running',
    pid = ?,
    started_at = strftime('%Y-%m-%dT%H:%M:%SZ', 'now')
WHERE id = ?;
```

**on-complete.py updates task result:**

```sql
UPDATE tasks
SET status = ?,               -- 'done' or 'failed'
    exit_code = ?,
    result_file = ?,
    result_summary = ?,
    cost_usd = ?,
    duration_ms = ?,
    num_turns = ?,
    error_message = ?,
    completed_at = strftime('%Y-%m-%dT%H:%M:%SZ', 'now')
WHERE session_id = ? AND status = 'running';

-- Also update agent state back to idle
UPDATE agents
SET state = 'idle',
    total_tasks = total_tasks + 1,
    updated_at = strftime('%Y-%m-%dT%H:%M:%SZ', 'now')
WHERE session_id = ?;
```

**Get unreported completions (Bridge Bot polls this):**

```sql
SELECT t.id, t.session_id, t.prompt, t.status, t.result_summary,
       t.cost_usd, t.duration_ms, t.error_message, t.completed_at,
       a.name AS agent_name
FROM tasks t
JOIN agents a ON t.session_id = a.session_id
WHERE t.reported = 0
  AND t.status IN ('done', 'failed', 'timeout', 'killed')
ORDER BY t.completed_at;
```

**Mark task as reported:**

```sql
UPDATE tasks SET reported = 1 WHERE id = ?;
```

**Get task history for an agent:**

```sql
SELECT id, prompt, status, result_summary, cost_usd, duration_ms,
       created_at, completed_at
FROM tasks
WHERE session_id = ?
ORDER BY created_at DESC
LIMIT ?;
```

**Watcher: find stale running tasks:**

```sql
SELECT id, session_id, pid, started_at
FROM tasks
WHERE status = 'running';
-- Application then checks os.kill(pid, 0) and compares started_at to now.
```

## 3. Workspace Structure

Each session gets a workspace directory under `~/.claude-bridge/workspaces/`. Bridge
owns this directory entirely -- Claude Code never reads from or writes to it.

```
~/.claude-bridge/workspaces/
    |
    +-- backend--my-api/
    |   |
    |   +-- metadata.json              Session creation metadata
    |   |
    |   +-- tasks/
    |       +-- task-1-result.json     Claude JSON output (full)
    |       +-- task-1-stderr.log      Captured stderr from subprocess
    |       +-- task-2-result.json
    |       +-- task-2-stderr.log
    |       +-- ...
    |
    +-- frontend--my-web/
    |   +-- metadata.json
    |   +-- tasks/
    |       +-- ...
    |
    +-- devops--infra/
        +-- metadata.json
        +-- tasks/
            +-- ...
```

### metadata.json

Created once when the agent is created. Never modified after that.

```json
{
  "session_id": "backend--my-api",
  "agent_name": "backend",
  "project_dir": "/Users/hieutran/projects/my-api",
  "purpose": "REST API development, database migrations, testing",
  "created_at": "2026-03-27T10:30:00Z",
  "bridge_version": "0.1.0"
}
```

### Task Result Files

**task-{id}-result.json** -- The full JSON output from `claude --output-format json`.
on-complete.py reads this to extract the summary, cost, turn count, and error status.

```json
{
  "type": "result",
  "subtype": "success",
  "is_error": false,
  "duration_ms": 45230,
  "duration_api_ms": 38100,
  "num_turns": 7,
  "result": "Added pagination to the /users endpoint...",
  "session_id": "backend--my-api",
  "cost_usd": 0.042,
  "total_cost_usd": 0.042
}
```

**task-{id}-stderr.log** -- Raw stderr captured from the claude subprocess. Useful for
debugging when the stop hook does not fire or the process crashes.

```
Loading agent: bridge--backend--my-api
Session: backend--my-api
Working directory: /Users/hieutran/projects/my-api
...
```

### Cleanup Strategy

To prevent unbounded disk growth:

- **Keep last N results per session** (default N=50, configurable in config.yaml).
- On each new task dispatch, check count and delete oldest result + stderr pairs
  that exceed the limit.
- Deletion is best-effort -- if it fails, the task still runs.
- metadata.json is never deleted (only removed on agent deletion).
- SQLite rows are retained indefinitely for history queries. Only the result files
  on disk are cleaned up. The `result_file` column becomes a dead pointer for old
  tasks, which is acceptable since the `result_summary` column has the essential
  information.

## 4. Agent .md File Format

Bridge generates agent files in the native Claude Code agent format. These live in
`~/.claude/agents/` which is where Claude Code looks for `--agent` definitions.

File naming: `bridge--{session_id}.md`

The `bridge--` prefix serves two purposes: (1) namespacing to avoid collisions with
user-created agents, and (2) making it easy to identify and clean up Bridge-managed
agents.

### Full Example

File: `~/.claude/agents/bridge--backend--my-api.md`

```markdown
---
name: bridge--backend--my-api
description: "Backend API development -- REST endpoints, database migrations, testing"
tools:
  - Read
  - Edit
  - Write
  - Bash
  - Grep
  - Glob
  - WebFetch
model: sonnet
isolation: worktree
memory: project
hooks:
  Stop:
    - hooks:
        - type: command
          command: "python3 ~/.claude-bridge/on-complete.py --session-id backend--my-api --task-id $TASK_ID"
---

# Agent: backend
**Project:** /Users/hieutran/projects/my-api
**Purpose:** REST API development, database migrations, testing

## Working Style
- Complete the task fully before stopping. Do not ask clarifying questions -- make
  reasonable assumptions and document them in your summary.
- Run the project's test suite after making changes, if one exists.
- Run the linter if configured, and fix any violations you introduce.
- When done, provide a clear summary of what you changed and why.
- If you encounter an error you cannot resolve, describe what you tried and what
  failed.

## Scope
- You are a backend engineer working on this project.
- Focus on server-side code, APIs, database queries, and backend tests.
- Do not modify frontend files, CI/CD pipelines, or infrastructure unless the task
  explicitly requires it.
```

### Frontmatter Fields

| Field         | Type          | Description |
|---------------|---------------|-------------|
| `name`        | string        | Must match the filename (without `.md`). Used by `--agent` flag. |
| `description` | string        | Human-readable one-liner. Shown in `claude --agent-list`. |
| `tools`       | list[string]  | Claude Code tools this agent is allowed to use. |
| `model`       | string        | Model to use: `sonnet`, `opus`, `haiku`. |
| `isolation`   | string        | `worktree` for git-based isolation. Each task gets a fresh worktree. |
| `memory`      | string        | `project` enables Auto Memory for the project directory. |
| `hooks`       | object        | Claude Code hook definitions. Bridge uses `Stop` to detect completion. |

### Hooks Detail

The `Stop` hook fires whenever the Claude Code session ends (success or failure).
Bridge uses it to run `on-complete.py`, which:

1. Reads the task result JSON from the workspace.
2. Extracts summary, cost, duration, turn count, and error status.
3. Updates the tasks table in SQLite.
4. Sets the agent state back to `idle`.
5. Prints a completion report (which the Bridge Bot session picks up via stdout).

The `$TASK_ID` variable is set as an environment variable by bridge-cli.py when
spawning the subprocess.

### Markdown Body (System Prompt)

The markdown below the frontmatter `---` fence becomes the agent's system prompt. It
is loaded into context at the start of every task and survives compaction. This is
where Bridge injects:

- The agent's role identity (backend, frontend, devops, etc.)
- The project directory path
- The stated purpose
- Working style instructions (complete tasks, run tests, summarize)
- Scope boundaries (what the agent should and should not touch)

## 5. Configuration

Global configuration lives at `~/.claude-bridge/config.yaml`. This file is for
Bridge daemon settings only. Telegram bot configuration is managed by the Telegram
MCP plugin and is not duplicated here.

```yaml
# ~/.claude-bridge/config.yaml

# Database
db_path: "~/.claude-bridge/bridge.db"

# Task execution
task_timeout_minutes: 30        # Watcher kills tasks running longer than this
max_results_per_session: 50     # Keep last N result files per session workspace

# Watcher (fallback completion detector)
watcher_interval_minutes: 5     # How often the cron watcher runs

# Logging
log_level: "INFO"               # DEBUG, INFO, WARNING, ERROR
log_file: "~/.claude-bridge/bridge.log"

# Claude Code defaults (can be overridden per agent .md)
default_model: "sonnet"
default_isolation: "worktree"
```

### What is NOT in config.yaml

- **Telegram bot token and chat ID** -- These are configured in the Telegram MCP
  plugin settings within Claude Code. Bridge never touches these directly.
- **API keys** -- Claude Code manages its own authentication. Bridge just wraps the
  CLI.
- **Per-agent settings** -- These live in the agent .md files, not in global config.

## 6. File Ownership Boundaries

Clear separation of what Bridge creates and manages vs what Claude Code creates and
manages. This boundary is critical -- crossing it leads to bugs and broken state.

```
+----------------------------------+-----------------------------------+
|  BRIDGE CREATES & MANAGES        |  CLAUDE CODE CREATES & MANAGES    |
|  (we write, we read)             |  (CC writes, we may read)         |
+----------------------------------+-----------------------------------+
|                                  |                                   |
|  ~/.claude-bridge/               |  ~/.claude/projects/              |
|    bridge.db                     |    <encoded-path>/memory/         |
|    config.yaml                   |      MEMORY.md                    |
|    bridge.log                    |      <topic>.md                   |
|    bridge-cli.py                 |                                   |
|    on-complete.py                |  ~/.claude/sessions/              |
|    watcher.py                    |    <session-id>.jsonl             |
|                                  |    (conversation history)         |
|  ~/.claude-bridge/workspaces/    |                                   |
|    {session_id}/                 |  <project>/.claude/worktrees/     |
|      metadata.json               |    <worktree-name>/               |
|      tasks/                      |    (isolated git copies)          |
|        task-{id}-result.json     |                                   |
|        task-{id}-stderr.log      |                                   |
|                                  |                                   |
+----------------------------------+-----------------------------------+
|  BRIDGE GENERATES                |  CLAUDE CODE GENERATES            |
|  (we write, CC reads)            |  (CC writes, we read)             |
+----------------------------------+-----------------------------------+
|                                  |                                   |
|  ~/.claude/agents/               |  Auto Memory files                |
|    bridge--{session_id}.md       |    ~/.claude/projects/*/memory/   |
|    (agent role definitions)      |    (learned patterns, prefs)      |
|                                  |                                   |
|  <project>/CLAUDE.md             |  Session JSONL files              |
|    (project context, generated   |    ~/.claude/sessions/*.jsonl     |
|     on agent creation via a      |    (full conversation logs)       |
|     one-shot Claude Code scan)   |                                   |
|                                  |  Worktree contents                |
|                                  |    Code changes made during tasks |
|                                  |                                   |
+----------------------------------+-----------------------------------+
```

### Rules

1. **Bridge never writes to `~/.claude/projects/`** -- Auto Memory is Claude Code's
   domain. Bridge reads MEMORY.md for the `/memory` command but never modifies it.
2. **Bridge never writes to `~/.claude/sessions/`** -- Session JSONL is Claude Code's
   internal format. Bridge has no reason to read or write these.
3. **Bridge never modifies worktree contents** -- Claude Code manages git worktrees.
   Bridge only passes the `isolation: worktree` flag in the agent .md.
4. **Claude Code never reads from `~/.claude-bridge/`** -- The Bridge workspace is
   invisible to Claude Code. The only touchpoint is the agent .md file (in
   `~/.claude/agents/`) and the project CLAUDE.md.
5. **CLAUDE.md is a one-time handoff** -- Bridge generates it during agent creation
   (via a one-shot scan). After that, it belongs to the project and both Bridge and
   Claude Code treat it as read-only.

## 7. Data Lifecycle

### 7.1 Agent Creation

Triggered by: `/create-agent backend /Users/hieutran/projects/my-api "REST API dev"`

```
Step  Action                             Artifact Created
----  ------                             ----------------
1     Validate project_dir exists         --
2     Derive session_id                   session_id = "backend--my-api"
3     Check uniqueness in SQLite          --
4     Generate agent .md file             ~/.claude/agents/bridge--backend--my-api.md
5     Create workspace directory          ~/.claude-bridge/workspaces/backend--my-api/
6     Write metadata.json                 ~/.claude-bridge/workspaces/backend--my-api/metadata.json
7     Create tasks/ subdirectory          ~/.claude-bridge/workspaces/backend--my-api/tasks/
8     Run CLAUDE.md init (one-shot scan)  /Users/hieutran/projects/my-api/CLAUDE.md
9     Insert row into agents table        bridge.db -> agents
```

CLAUDE.md init (step 8) spawns a one-shot Claude Code process:

```bash
claude -p "Scan this project and generate a CLAUDE.md. Include: project overview,
build/test/lint commands, directory structure, coding conventions. Also note that
this project will be worked on by a Bridge agent with purpose: REST API development."
```

If CLAUDE.md already exists, the init appends an agent-context section rather than
overwriting.

### 7.2 Task Dispatch

Triggered by: `/task backend add pagination to /users`

```
Step  Action                             State Change
----  ------                             ------------
1     Look up agent by name              SELECT from agents WHERE name = 'backend'
2     Check agent not busy               SELECT from tasks WHERE session_id = ? AND status = 'running'
3     Insert task row (pending)          tasks: id=1, status='pending'
4     Set agent state to busy            agents: state='busy'
5     Spawn subprocess                   claude --agent ... --session-id ... -p "..."
6     Record PID                         tasks: pid=12345, status='running', started_at=now
7     Redirect stdout to result file     workspaces/backend--my-api/tasks/task-1-result.json
8     Redirect stderr to log file        workspaces/backend--my-api/tasks/task-1-stderr.log
9     Return confirmation                "Task #1 dispatched to backend"
```

The subprocess is fire-and-forget from bridge-cli.py's perspective. The stop hook
handles completion.

### 7.3 Task Completion

Triggered by: Claude Code finishing the task (stop hook fires)

```
Step  Action                             State Change
----  ------                             ------------
1     Stop hook calls on-complete.py      --
2     Read result JSON from result file   Parse task-1-result.json
3     Extract fields                      summary, cost, duration, turns, is_error
4     Update task row                     tasks: status='done', result_summary=..., cost_usd=...
5     Update agent state                  agents: state='idle', total_tasks += 1
6     Print report to stdout              Bridge Bot picks this up from the session
7     Bridge Bot marks reported           tasks: reported=1
```

If the stop hook does not fire (process crashed, SIGKILL, etc.), the watcher catches
it:

```
Step  Action                             State Change
----  ------                             ------------
1     Cron runs watcher.py every 5 min    --
2     Query running tasks                 SELECT from tasks WHERE status = 'running'
3     For each: check os.kill(pid, 0)     --
4a    If process dead + no result:        tasks: status='failed', error_message='Process died'
4b    If running > timeout:               SIGTERM -> tasks: status='timeout'
5     Update agent state                  agents: state='idle'
```

### 7.4 Agent Deletion

Triggered by: `/delete-agent backend`

```
Step  Action                             Artifact Removed
----  ------                             ----------------
1     Check for running task              --
2     If running: SIGTERM the process     Kill PID
3     Wait briefly for graceful shutdown  --
4     Delete SQLite rows                  DELETE FROM agents WHERE name = ? (CASCADE deletes tasks)
5     Delete workspace directory          rm -rf ~/.claude-bridge/workspaces/backend--my-api/
6     Delete agent .md file               rm ~/.claude/agents/bridge--backend--my-api.md
7     (Do NOT delete Auto Memory)         ~/.claude/projects/ is untouched
8     (Do NOT delete CLAUDE.md)           Project CLAUDE.md is untouched
```

Auto Memory and CLAUDE.md are intentionally preserved. They contain project knowledge
that is useful regardless of whether a Bridge agent exists, and they may be shared
with other agents or manual Claude Code sessions.

## 8. Auto Memory Integration

Claude Code automatically creates and maintains memory files for each project. Bridge
reads these for the `/memory` command but never writes to them.

### How Auto Memory Works

When Claude Code runs with `memory: project` enabled (set in the agent .md
frontmatter), it automatically:

1. Creates `~/.claude/projects/<encoded-path>/memory/MEMORY.md` after significant
   interactions.
2. Adds topic-specific files like `api_patterns.md`, `testing.md`, etc.
3. Updates these files as the agent learns more about the project.

### Path Encoding

Claude Code encodes project paths as directory names by replacing `/` with `-` and
prepending a `-`. The encoding turns an absolute path into a flat directory name.

```
Project path:    /Users/hieutran/projects/my-api
Encoded:         -Users-hieutran-projects-my-api
Memory location: ~/.claude/projects/-Users-hieutran-projects-my-api/memory/
```

Bridge must replicate this encoding to find the right memory directory. The algorithm:

```python
import os

def get_auto_memory_dir(project_dir: str) -> str:
    """Get the Auto Memory directory for a project path."""
    # Claude Code encodes paths by replacing / with - and prepending -
    abs_path = os.path.abspath(project_dir)
    encoded = abs_path.replace("/", "-")
    # The result already starts with - because abs_path starts with /
    return os.path.expanduser(f"~/.claude/projects/{encoded}/memory/")
```

### What Bridge Reads

For the `/memory` command, Bridge reads:

1. **MEMORY.md** -- The main memory file with project-level observations.
2. **Topic files** -- Any additional `.md` files in the memory directory.

```python
def read_agent_memory(session_id: str) -> dict:
    """Read all Auto Memory files for a session's project."""
    agent = db.get_agent_by_session(session_id)
    memory_dir = get_auto_memory_dir(agent["project_dir"])

    result = {"main": None, "topics": {}}

    memory_file = os.path.join(memory_dir, "MEMORY.md")
    if os.path.exists(memory_file):
        with open(memory_file) as f:
            result["main"] = f.read()

    for fname in os.listdir(memory_dir):
        if fname.endswith(".md") and fname != "MEMORY.md":
            topic = fname.removesuffix(".md")
            with open(os.path.join(memory_dir, fname)) as f:
                result["topics"][topic] = f.read()

    return result
```

### Read-Only Contract

Bridge treats Auto Memory as strictly read-only:

- Bridge never creates files in `~/.claude/projects/`.
- Bridge never modifies existing memory files.
- Bridge never deletes memory files (even during agent deletion).
- If the memory directory does not exist yet (agent has not run any tasks), Bridge
  reports "No memory yet" rather than creating the directory.

This is important because Claude Code manages memory file format and content. If
Bridge were to write to these files, it could corrupt Claude Code's internal
expectations or cause merge conflicts in the memory content.
