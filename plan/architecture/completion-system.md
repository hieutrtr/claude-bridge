# Task Completion System Architecture

> How Claude Bridge detects task completion, captures results, updates state, and reports back to the user.

---

## 1. Overview

The completion system uses a **dual mechanism** to guarantee that every task is accounted for:

```
                   Task finishes
                        |
            +-----------+-----------+
            |                       |
            v                       v
    +---------------+       +---------------+
    |  Stop Hook    |       |  PID Watcher  |
    |  (primary)    |       |  (fallback)   |
    |               |       |               |
    |  Immediate    |       |  Cron: */5min |
    |  on-complete  |       |  watcher.py   |
    |  .py fires    |       |  checks PIDs  |
    +-------+-------+       +-------+-------+
            |                       |
            v                       v
    +---------------------------------------+
    |          SQLite (bridge.db)            |
    |  task status, result, cost, duration   |
    |  agent state -> idle                  |
    +-------------------+-------------------+
                        |
                        v
    +---------------------------------------+
    |        Report to Telegram             |
    |  Success/failure message to user      |
    +---------------------------------------+
```

**Primary (Stop Hook):** Claude Code natively fires a Stop hook when any agent task finishes, whether successfully or with an error. The hook calls `on-complete.py`, which parses the result, updates SQLite, and prints a formatted report. This covers ~99% of completions with zero latency.

**Fallback (PID Watcher):** A cron job runs `watcher.py` every 5 minutes. It queries for tasks still marked as `running`, checks whether their PIDs are alive, and handles orphaned or timed-out tasks. This catches edge cases where the Stop hook never fires: SIGKILL, process crashes, or hook script errors.

No polling is needed for normal operation. The Stop hook is event-driven and fires immediately.

---

## 2. Stop Hook (Primary)

### 2.1 How It Works

Each agent `.md` file generated by Bridge includes a Stop hook in its YAML frontmatter:

```yaml
---
name: bridge--backend--my-api
description: "API dev, REST endpoints, DB migrations"
tools: Read, Edit, Write, Bash, Grep, Glob
model: sonnet
isolation: worktree
memory: project
hooks:
  Stop:
    - type: command
      command: "python3 ~/.claude-bridge/on-complete.py --session-id backend--my-api"
---
```

When Claude Code finishes executing a task through this agent (regardless of success or failure), it automatically fires every hook registered under `Stop`. Claude Code invokes the command as a subprocess, passing result data through the execution environment.

### 2.2 Trigger Conditions

The Stop hook fires in all of these scenarios:

| Scenario | Hook Fires? | `is_error` |
|---|---|---|
| Task completes successfully | Yes | `false` |
| Task fails (code error, tool error) | Yes | `true` |
| User cancels via Ctrl-C (interactive) | Yes | `true` |
| Process receives SIGTERM | Yes (usually) | `true` |
| Process receives SIGKILL | **No** | N/A |
| Claude Code crashes / segfault | **No** | N/A |
| Hook script itself errors | N/A (partial) | N/A |

The last three cases are why the fallback watcher exists.

### 2.3 on-complete.py Receives Session ID

The `--session-id` argument is baked into the agent `.md` at creation time. When the hook fires, `on-complete.py` uses this to:

1. Locate the correct task in SQLite (the most recent `running` task for that session)
2. Find the result file in the session's workspace directory
3. Update the correct agent record back to `idle`

### 2.4 on-complete.py Full Logic

```python
#!/usr/bin/env python3
"""
Stop hook handler for Claude Bridge.
Called by Claude Code when an agent task finishes.

Usage: python3 on-complete.py --session-id <session_id>
"""

import argparse
import json
import os
import sqlite3
import sys
import time
from datetime import datetime, timezone
from pathlib import Path

BRIDGE_HOME = Path.home() / ".claude-bridge"
DB_PATH = BRIDGE_HOME / "bridge.db"
SUMMARY_MAX_LEN = 500


def parse_args():
    parser = argparse.ArgumentParser()
    parser.add_argument("--session-id", required=True)
    return parser.parse_args()


def get_running_task(db, session_id):
    """Find the most recent running task for this session."""
    cursor = db.execute(
        """
        SELECT id, pid, result_file
        FROM tasks
        WHERE session_id = ? AND status = 'running'
        ORDER BY created_at DESC
        LIMIT 1
        """,
        (session_id,),
    )
    row = cursor.fetchone()
    if not row:
        return None
    return {"id": row[0], "pid": row[1], "result_file": row[2]}


def parse_result_file(result_file_path):
    """Parse the JSON result file written by claude --output-format json."""
    path = Path(result_file_path).expanduser()
    if not path.exists():
        return {
            "type": "result",
            "is_error": True,
            "result": "Result file not found",
            "cost_usd": 0.0,
            "duration_ms": 0,
            "num_turns": 0,
            "session_id": None,
        }

    try:
        with open(path) as f:
            data = json.load(f)
    except json.JSONDecodeError:
        return {
            "type": "result",
            "is_error": True,
            "result": "Invalid JSON in result file",
            "cost_usd": 0.0,
            "duration_ms": 0,
            "num_turns": 0,
            "session_id": None,
        }

    return data


def truncate_summary(text, max_len=SUMMARY_MAX_LEN):
    """Truncate result text to max_len for SQLite storage."""
    if not text:
        return ""
    text = str(text)
    if len(text) <= max_len:
        return text
    return text[: max_len - 3] + "..."


def update_task(db, task_id, result_data):
    """Update the task row with completion data."""
    is_error = result_data.get("is_error", False)
    status = "failed" if is_error else "done"
    result_text = result_data.get("result", "")

    # result can be a string or a list of content blocks
    if isinstance(result_text, list):
        # Extract text from content blocks
        parts = []
        for block in result_text:
            if isinstance(block, dict) and block.get("type") == "text":
                parts.append(block.get("text", ""))
            elif isinstance(block, str):
                parts.append(block)
        result_text = "\n".join(parts)

    summary = truncate_summary(result_text)
    cost_usd = result_data.get("cost_usd", 0.0)
    duration_ms = result_data.get("duration_ms", 0)
    num_turns = result_data.get("num_turns", 0)
    exit_code = 1 if is_error else 0
    error_message = result_text if is_error else None
    completed_at = datetime.now(timezone.utc).isoformat()

    db.execute(
        """
        UPDATE tasks SET
            status = ?,
            result_summary = ?,
            cost_usd = ?,
            duration_ms = ?,
            num_turns = ?,
            exit_code = ?,
            error_message = ?,
            completed_at = ?
        WHERE id = ?
        """,
        (
            status,
            summary,
            cost_usd,
            duration_ms,
            num_turns,
            exit_code,
            error_message,
            completed_at,
            task_id,
        ),
    )

    return status, summary, cost_usd, duration_ms


def update_agent(db, session_id):
    """Mark the agent as idle and bump task counters."""
    now = datetime.now(timezone.utc).isoformat()
    db.execute(
        """
        UPDATE agents SET
            state = 'idle',
            last_task_at = ?,
            total_tasks = total_tasks + 1
        WHERE session_id = ?
        """,
        (now, session_id),
    )


def print_report(task_id, status, summary, cost_usd, duration_ms, session_id):
    """Print a formatted report to stdout.

    The Bridge Bot picks up this stdout output to relay to Telegram.
    """
    duration_sec = (duration_ms or 0) / 1000
    cost_str = f"${cost_usd:.4f}" if cost_usd else "$0.00"

    if duration_sec >= 60:
        duration_str = f"{duration_sec / 60:.1f}m"
    else:
        duration_str = f"{duration_sec:.0f}s"

    agent_name = session_id.split("--")[0] if "--" in session_id else session_id

    if status == "done":
        print(f"Task #{task_id} ({agent_name}) done in {duration_str}. "
              f"Cost: {cost_str}. {summary}")
    else:
        print(f"Task #{task_id} ({agent_name}) failed after {duration_str}. "
              f"Cost: {cost_str}. {summary}")


def main():
    args = parse_args()
    session_id = args.session_id

    db = sqlite3.connect(str(DB_PATH), timeout=10)
    db.execute("PRAGMA journal_mode=WAL")
    try:
        task = get_running_task(db, session_id)
        if not task:
            print(f"No running task found for session {session_id}", file=sys.stderr)
            return 1

        task_id = task["id"]
        result_file = task["result_file"]

        # Parse result
        result_data = parse_result_file(result_file)

        # Update task
        status, summary, cost_usd, duration_ms = update_task(db, task_id, result_data)

        # Update agent
        update_agent(db, session_id)

        db.commit()

        # Print report (Bridge Bot captures this)
        print_report(task_id, status, summary, cost_usd, duration_ms, session_id)

        return 0

    except Exception as e:
        print(f"on-complete.py error: {e}", file=sys.stderr)
        # Even if we fail, try to at least mark the task
        try:
            db.execute(
                "UPDATE tasks SET status = 'failed', error_message = ? WHERE id = ?",
                (str(e), task["id"]),
            )
            db.execute(
                "UPDATE agents SET state = 'idle' WHERE session_id = ?",
                (session_id,),
            )
            db.commit()
        except Exception:
            pass
        return 1
    finally:
        db.close()


if __name__ == "__main__":
    sys.exit(main())
```

### 2.5 Report Formats

**Success:**
```
Task #3 (backend) done in 2.4m. Cost: $0.0412. Added pagination to /users endpoint with cursor-based navigation. Updated tests and migration.
```

**Failure:**
```
Task #5 (backend) failed after 1.1m. Cost: $0.0187. Could not resolve import for @prisma/client. Check that dependencies are installed.
```

### 2.6 How the Report Reaches Telegram

`on-complete.py` prints its report to stdout. The delivery path works as follows:

1. Claude Code fires the Stop hook as a subprocess
2. `on-complete.py` prints the formatted report to stdout
3. The Bridge Bot queries SQLite for tasks where `reported = 0`
4. For each unreported task, it reads the `result_summary` from SQLite and sends it as a Telegram message
5. After sending, it sets `reported = 1`

This decoupled design means that even if stdout capture fails, the data is still in SQLite and the Bridge Bot will pick it up on its next check.

## 3. Watcher (Fallback)

### 3.1 When the Stop Hook Does Not Fire

The Stop hook will not fire in these situations:

- **SIGKILL:** The process is force-killed (`kill -9`). The OS terminates it immediately with no opportunity to run cleanup or hooks.
- **Process crash:** A segfault, out-of-memory kill, or other hard crash prevents orderly shutdown.
- **Hook script error:** If `on-complete.py` itself crashes before updating SQLite (e.g., disk full, permissions error), the task remains in `running` state.
- **System shutdown/restart:** The machine powers off or restarts while a task is running.

### 3.2 Cron Schedule

The watcher runs every 5 minutes via cron:

```
*/5 * * * * python3 ~/.claude-bridge/watcher.py >> ~/.claude-bridge/watcher.log 2>&1
```

This means the worst-case detection latency for an orphaned task is 5 minutes. For normal completions (Stop hook), latency is effectively zero.

### 3.3 watcher.py Full Logic

```python
#!/usr/bin/env python3
"""
Fallback watcher for Claude Bridge.
Detects orphaned tasks where the Stop hook did not fire.
Runs via cron every 5 minutes.

Crontab: */5 * * * * python3 ~/.claude-bridge/watcher.py
"""

import json
import os
import signal
import sqlite3
import sys
import time
from datetime import datetime, timezone
from pathlib import Path

BRIDGE_HOME = Path.home() / ".claude-bridge"
DB_PATH = BRIDGE_HOME / "bridge.db"
TIMEOUT_MINUTES = 30
SUMMARY_MAX_LEN = 500


def is_pid_alive(pid):
    """Check if a process with the given PID is still running."""
    if not pid:
        return False
    try:
        os.kill(pid, 0)  # Signal 0 = existence check, no actual signal sent
        return True
    except ProcessLookupError:
        return False  # No such process
    except PermissionError:
        return True  # Process exists but we lack permission (still alive)


def get_running_tasks(db):
    """Query all tasks currently marked as running."""
    cursor = db.execute(
        """
        SELECT id, session_id, pid, result_file, created_at
        FROM tasks
        WHERE status = 'running'
        """
    )
    columns = ["id", "session_id", "pid", "result_file", "created_at"]
    return [dict(zip(columns, row)) for row in cursor.fetchall()]


def get_unreported_tasks(db):
    """Query completed tasks that have not been reported to Telegram."""
    cursor = db.execute(
        """
        SELECT id, session_id, status, result_summary, cost_usd, duration_ms
        FROM tasks
        WHERE status IN ('done', 'failed', 'timeout')
          AND reported = 0
        """
    )
    columns = ["id", "session_id", "status", "result_summary", "cost_usd", "duration_ms"]
    return [dict(zip(columns, row)) for row in cursor.fetchall()]


def parse_result_file(result_file_path):
    """Attempt to read and parse a result file."""
    if not result_file_path:
        return None
    path = Path(result_file_path).expanduser()
    if not path.exists():
        return None
    try:
        with open(path) as f:
            return json.load(f)
    except (json.JSONDecodeError, OSError):
        return None


def truncate(text, max_len=SUMMARY_MAX_LEN):
    if not text:
        return ""
    text = str(text)
    return text if len(text) <= max_len else text[: max_len - 3] + "..."


def task_age_minutes(created_at_str):
    """Calculate how many minutes old a task is."""
    try:
        created = datetime.fromisoformat(created_at_str)
        if created.tzinfo is None:
            created = created.replace(tzinfo=timezone.utc)
        now = datetime.now(timezone.utc)
        return (now - created).total_seconds() / 60
    except (ValueError, TypeError):
        return 0


def handle_dead_process(db, task):
    """Handle a task whose PID is no longer alive."""
    task_id = task["id"]
    session_id = task["session_id"]
    now = datetime.now(timezone.utc).isoformat()

    # Try to salvage result from result file
    result_data = parse_result_file(task["result_file"])

    if result_data:
        is_error = result_data.get("is_error", False)
        status = "failed" if is_error else "done"
        result_text = result_data.get("result", "")
        if isinstance(result_text, list):
            parts = []
            for block in result_text:
                if isinstance(block, dict) and block.get("type") == "text":
                    parts.append(block.get("text", ""))
                elif isinstance(block, str):
                    parts.append(block)
            result_text = "\n".join(parts)
        summary = truncate(result_text)
        cost_usd = result_data.get("cost_usd", 0.0)
        duration_ms = result_data.get("duration_ms", 0)
        num_turns = result_data.get("num_turns", 0)
    else:
        # No result file -- process died without writing output
        status = "failed"
        summary = "Process terminated without writing result (possible crash or SIGKILL)"
        cost_usd = 0.0
        duration_ms = 0
        num_turns = 0

    db.execute(
        """
        UPDATE tasks SET
            status = ?,
            result_summary = ?,
            cost_usd = ?,
            duration_ms = ?,
            num_turns = ?,
            exit_code = ?,
            error_message = ?,
            completed_at = ?
        WHERE id = ?
        """,
        (
            status,
            summary,
            cost_usd,
            duration_ms,
            num_turns,
            1 if status == "failed" else 0,
            summary if status == "failed" else None,
            now,
            task_id,
        ),
    )

    # Mark agent idle
    db.execute(
        """
        UPDATE agents SET
            state = 'idle',
            last_task_at = ?,
            total_tasks = total_tasks + 1
        WHERE session_id = ?
        """,
        (now, session_id),
    )

    print(f"[watcher] Task #{task_id} ({session_id}): PID dead, marked as {status}")


def handle_timeout(db, task):
    """Kill a task that has been running longer than TIMEOUT_MINUTES."""
    task_id = task["id"]
    session_id = task["session_id"]
    pid = task["pid"]
    now = datetime.now(timezone.utc).isoformat()

    # Send SIGTERM first, then SIGKILL if needed
    try:
        os.kill(pid, signal.SIGTERM)
        time.sleep(2)
        if is_pid_alive(pid):
            os.kill(pid, signal.SIGKILL)
    except ProcessLookupError:
        pass  # Already dead

    age = task_age_minutes(task["created_at"])
    summary = f"Task timed out after {age:.0f} minutes (limit: {TIMEOUT_MINUTES}m)"

    db.execute(
        """
        UPDATE tasks SET
            status = 'timeout',
            result_summary = ?,
            exit_code = -1,
            error_message = ?,
            completed_at = ?
        WHERE id = ?
        """,
        (summary, summary, now, task_id),
    )

    db.execute(
        """
        UPDATE agents SET
            state = 'idle',
            last_task_at = ?,
            total_tasks = total_tasks + 1
        WHERE session_id = ?
        """,
        (now, session_id),
    )

    print(f"[watcher] Task #{task_id} ({session_id}): timed out after {age:.0f}m, killed")


def report_unreported(db):
    """Print reports for any completed tasks that haven't been reported yet.

    The Bridge Bot picks these up and forwards to Telegram.
    """
    tasks = get_unreported_tasks(db)
    for task in tasks:
        task_id = task["id"]
        session_id = task["session_id"]
        status = task["status"]
        summary = task["result_summary"] or ""
        cost_usd = task["cost_usd"] or 0
        duration_ms = task["duration_ms"] or 0

        duration_sec = duration_ms / 1000
        if duration_sec >= 60:
            duration_str = f"{duration_sec / 60:.1f}m"
        else:
            duration_str = f"{duration_sec:.0f}s"

        agent_name = session_id.split("--")[0] if "--" in session_id else session_id
        cost_str = f"${cost_usd:.4f}" if cost_usd else "$0.00"

        if status == "done":
            print(f"Task #{task_id} ({agent_name}) done in {duration_str}. "
                  f"Cost: {cost_str}. {summary}")
        elif status == "timeout":
            print(f"Task #{task_id} ({agent_name}) timed out. "
                  f"Cost: {cost_str}. {summary}")
        else:
            print(f"Task #{task_id} ({agent_name}) failed after {duration_str}. "
                  f"Cost: {cost_str}. {summary}")

        db.execute("UPDATE tasks SET reported = 1 WHERE id = ?", (task_id,))

    return len(tasks)


def main():
    db = sqlite3.connect(str(DB_PATH), timeout=10)
    db.execute("PRAGMA journal_mode=WAL")

    try:
        running_tasks = get_running_tasks(db)

        for task in running_tasks:
            pid = task["pid"]
            age = task_age_minutes(task["created_at"])

            if not is_pid_alive(pid):
                # Process is dead but task is still marked running
                # The Stop hook must have failed to fire
                handle_dead_process(db, task)

            elif age > TIMEOUT_MINUTES:
                # Process is alive but has exceeded the time limit
                handle_timeout(db, task)

            # else: process is alive and within time limit -- do nothing

        # Report any completions that haven't been sent to Telegram
        unreported_count = report_unreported(db)

        db.commit()

        if running_tasks or unreported_count:
            print(f"[watcher] Checked {len(running_tasks)} running tasks, "
                  f"reported {unreported_count} unreported completions")

    except Exception as e:
        print(f"[watcher] Error: {e}", file=sys.stderr)
        return 1
    finally:
        db.close()

    return 0


if __name__ == "__main__":
    sys.exit(main())
```

### 3.4 Watcher Decision Matrix

```
Task in SQLite        PID alive?     Age > 30m?      Action
-----------------------------------------------------------------
status = 'running'    No             N/A             handle_dead_process()
                                                      Read result file if exists
                                                      Mark done or failed
                                                      Set agent idle

status = 'running'    Yes            No              Do nothing (task is active)

status = 'running'    Yes            Yes             handle_timeout()
                                                      SIGTERM -> SIGKILL
                                                      Mark timeout
                                                      Set agent idle

status = 'done'       N/A            N/A             Check reported flag
reported = 0                                          Print report for Bridge Bot

status = 'failed'     N/A            N/A             Check reported flag
reported = 0                                          Print report for Bridge Bot
```

## 4. Result File Parsing

### 4.1 Location

Each task has a dedicated result file at:

```
~/.claude-bridge/workspaces/{session_id}/tasks/task-{id}-result.json
```

For example:
```
~/.claude-bridge/workspaces/backend--my-api/tasks/task-3-result.json
```

This path is stored in the `result_file` column of the `tasks` table at dispatch time, before the Claude Code subprocess is launched. The subprocess is invoked with `--output-format json` and its stdout is redirected to this file.

### 4.2 JSON Format

Claude Code's `--output-format json` produces output with the following structure:

```json
{
  "type": "result",
  "subtype": "success",
  "is_error": false,
  "result": [
    {
      "type": "text",
      "text": "I've added pagination to the /users endpoint using cursor-based navigation. Changes:\n\n1. Added `cursor` and `limit` query params to GET /users\n2. Created `PaginationHelper` utility class\n3. Updated Prisma query to use cursor-based pagination\n4. Added tests for pagination edge cases\n\nAll 12 tests pass."
    }
  ],
  "cost_usd": 0.0412,
  "duration_ms": 144320,
  "num_turns": 8,
  "session_id": "abc123-def456"
}
```

**Field descriptions:**

| Field | Type | Description |
|---|---|---|
| `type` | string | Always `"result"` for final output |
| `subtype` | string | `"success"` or `"error_max_turns"` or other error subtypes |
| `is_error` | boolean | `true` if the task failed |
| `result` | string or array | The agent's final output. Can be a plain string or an array of content blocks |
| `cost_usd` | float | Total API cost for this task |
| `duration_ms` | integer | Wall-clock time in milliseconds |
| `num_turns` | integer | Number of agent turns (tool uses) |
| `session_id` | string | Claude Code's internal session ID |

### 4.3 Error Cases

**File missing:** The process died before writing output (SIGKILL, crash, OOM). The watcher detects a dead PID and marks the task as `failed` with a descriptive message.

```python
if not path.exists():
    return {
        "is_error": True,
        "result": "Result file not found",
        "cost_usd": 0.0,
        "duration_ms": 0,
        "num_turns": 0,
    }
```

**Invalid JSON:** The process was killed mid-write, producing a truncated file. Treated identically to a missing file.

```python
try:
    data = json.load(f)
except json.JSONDecodeError:
    return {
        "is_error": True,
        "result": "Invalid JSON in result file",
        ...
    }
```

**`is_error` is `true`:** The task ran to completion but the agent itself reported failure (e.g., could not resolve an import, tests failed, tool error). The result text contains the error details and is stored as-is.

### 4.4 Result Text Extraction

The `result` field can be either a plain string or an array of content blocks. Both forms must be handled:

```python
result_text = result_data.get("result", "")

if isinstance(result_text, list):
    parts = []
    for block in result_text:
        if isinstance(block, dict) and block.get("type") == "text":
            parts.append(block.get("text", ""))
        elif isinstance(block, str):
            parts.append(block)
    result_text = "\n".join(parts)
```

### 4.5 Truncation

The full result text can be lengthy (multiple paragraphs of agent output). For SQLite storage and Telegram delivery, the summary is truncated to 500 characters:

```python
SUMMARY_MAX_LEN = 500

def truncate_summary(text, max_len=SUMMARY_MAX_LEN):
    if not text:
        return ""
    text = str(text)
    if len(text) <= max_len:
        return text
    return text[:max_len - 3] + "..."
```

The full result remains available in the result JSON file on disk for inspection.

## 5. SQLite Updates

### 5.1 Task Update

When a task completes (via either Stop hook or watcher), the `tasks` row is updated:

```sql
UPDATE tasks SET
    status          = ?,     -- 'done', 'failed', 'timeout', 'killed'
    result_summary  = ?,     -- Truncated to 500 chars
    cost_usd        = ?,     -- e.g. 0.0412
    duration_ms     = ?,     -- e.g. 144320
    num_turns       = ?,     -- e.g. 8
    exit_code       = ?,     -- 0 = success, 1 = failure, -1 = timeout
    error_message   = ?,     -- NULL on success, error text on failure
    completed_at    = ?      -- ISO 8601 UTC timestamp
WHERE id = ?
```

### 5.2 Agent Update

The agent is marked idle so it can accept new tasks:

```sql
UPDATE agents SET
    state        = 'idle',           -- Was 'busy'
    last_task_at = ?,                -- ISO 8601 UTC timestamp
    total_tasks  = total_tasks + 1   -- Running counter
WHERE session_id = ?
```

### 5.3 Reported Flag

The `reported` column tracks whether a completion has been sent to Telegram:

```sql
-- After sending to Telegram:
UPDATE tasks SET reported = 1 WHERE id = ?
```

This ensures that even if the initial report delivery fails, the watcher will pick it up on its next run and attempt to report it again. A task is only marked `reported = 1` after the message has been successfully printed/sent.

### 5.4 WAL Mode for Concurrent Access

Both `on-complete.py` and `watcher.py` set WAL (Write-Ahead Logging) mode:

```python
db.execute("PRAGMA journal_mode=WAL")
```

WAL mode allows concurrent readers and a single writer without blocking. This is important because:

- The Bridge Bot may be reading task status while `on-complete.py` is writing
- The watcher may be updating tasks while the Bridge Bot is querying
- `bridge-cli.py` may be inserting a new task while `on-complete.py` is updating another

A 10-second connection timeout is used to handle brief lock contention:

```python
db = sqlite3.connect(str(DB_PATH), timeout=10)
```

## 6. Report Delivery to Telegram

There are two paths for getting completion reports to the user on Telegram:

### 6.1 Path A: Bridge Bot Queries SQLite (Primary)

The Bridge Bot (Session #0) periodically checks for unreported completions:

```
Bridge Bot (Claude Code + Telegram MCP)
    |
    |  1. Bash: python3 bridge-cli.py unreported
    |     -> Returns tasks where reported = 0
    |
    |  2. For each unreported task:
    |     Send Telegram message via MCP channel
    |
    |  3. Bash: python3 bridge-cli.py mark-reported --task-id N
    |     -> Sets reported = 1
    |
    v
User sees message on Telegram
```

The Bridge Bot can check for unreported tasks:

- On a regular interval (its own internal loop or prompted by CLAUDE.md instructions)
- Before processing any new user command (opportunistic check)
- When explicitly asked via `/status`

### 6.2 Path B: Watcher Reports (Fallback)

If a task was completed by the watcher (because the Stop hook failed), the watcher itself prints the report to stdout. Since the watcher runs via cron, this output goes to `~/.claude-bridge/watcher.log`. The Bridge Bot can be instructed to tail this log, or more practically, it simply queries for `reported = 0` tasks, which will include watcher-resolved tasks.

## 7. Edge Cases

### 7.1 Process Dies Before Writing Result File

**Scenario:** Claude Code receives SIGKILL or the machine loses power while a task is running. No result file is written.

**Detection:** The watcher finds `status = 'running'` in SQLite, calls `os.kill(pid, 0)`, gets `ProcessLookupError`.

**Resolution:** The watcher checks for the result file. If missing, the task is marked `failed` with the message `"Process terminated without writing result (possible crash or SIGKILL)"`. Cost and duration are recorded as zero.

**Risk:** PID reuse. If enough time passes, the OS may assign the same PID to a new process. Mitigation: the watcher runs every 5 minutes, and PID reuse on macOS requires cycling through the full PID range (typically 99999). The probability of a false positive within 5 minutes is negligible. An additional safeguard could check the process creation time (`ps -p PID -o lstart`) against the task's `created_at`.

### 7.2 Multiple Tasks Complete Simultaneously

**Scenario:** Two agents finish their tasks at the same moment. Two instances of `on-complete.py` run concurrently.

**Resolution:** Each instance targets a different session (the `--session-id` argument is unique per agent). They update different rows in different tables. SQLite WAL mode handles concurrent writes by serializing them. The 10-second timeout ensures neither instance fails due to a brief lock.

**Note:** In the current MVP design, each agent runs one task at a time, so true simultaneous completion within a single agent is not possible. Cross-agent concurrency is safe by design.

### 7.3 SQLite Locking During Concurrent Updates

**Scenario:** `on-complete.py`, `watcher.py`, and `bridge-cli.py` all try to write to SQLite at the same instant.

**Resolution:**

1. **WAL mode:** Allows concurrent reads with a single writer, dramatically reducing contention compared to the default rollback journal.
2. **Connection timeout:** `sqlite3.connect(path, timeout=10)` waits up to 10 seconds for a lock before raising `OperationalError`.
3. **Short transactions:** Each script opens a connection, performs its update (a few milliseconds of write time), commits, and closes. Lock hold times are sub-millisecond.
4. **Retry in watcher:** The watcher runs every 5 minutes. If it fails due to a lock timeout, it simply retries on the next cron invocation.

### 7.4 on-complete.py Crashes Mid-Update

**Scenario:** `on-complete.py` successfully parses the result file but crashes before committing the SQLite transaction (e.g., disk full, Python exception).

**Resolution:**

1. **SQLite atomicity:** If the process dies before `db.commit()`, the transaction is rolled back automatically. The task remains in `running` state.
2. **Watcher catches it:** On the next cron run (within 5 minutes), the watcher sees `status = 'running'` and `is_pid_alive()` returns `False`. It reads the result file (which was written successfully by Claude Code before the hook fired) and completes the update.
3. **Exception handler:** `on-complete.py` includes a catch-all exception handler that attempts a minimal update (mark task as failed, set agent idle) even when the primary logic fails.

```python
except Exception as e:
    print(f"on-complete.py error: {e}", file=sys.stderr)
    try:
        db.execute(
            "UPDATE tasks SET status = 'failed', error_message = ? WHERE id = ?",
            (str(e), task["id"]),
        )
        db.execute(
            "UPDATE agents SET state = 'idle' WHERE session_id = ?",
            (session_id,),
        )
        db.commit()
    except Exception:
        pass  # Watcher will handle it
```

### 7.5 Result File Written but Corrupt

**Scenario:** Claude Code is killed via SIGTERM during result file write. The file exists but contains truncated JSON.

**Resolution:** `json.load()` raises `JSONDecodeError`. The parser returns a synthetic error result. The task is marked as `failed` with `"Invalid JSON in result file"`. The watcher behaves identically if it encounters this case.

### 7.6 Agent Stuck in "Busy" State

**Scenario:** A task completes and both the Stop hook and watcher fail to update the agent (e.g., both crash, SQLite is locked for extended period). The agent remains in `state = 'busy'` and rejects new tasks.

**Resolution:** The `/kill` command in `bridge-cli.py` can force-reset an agent:

```
/kill backend
```

This sends SIGTERM to the PID (if alive), sets `status = 'killed'` on the task, and sets `state = 'idle'` on the agent. It is a manual override for cases where automated recovery fails.

### 7.7 Watcher and Stop Hook Both Fire

**Scenario:** The Stop hook fires but is slow (e.g., SQLite is temporarily locked). Before it commits, the watcher runs and also detects the task.

**Resolution:** This is a race condition that resolves safely:

1. If the watcher runs first and sees the PID as dead, it updates the task. When `on-complete.py` then tries to update, it finds no task with `status = 'running'` for that session and logs a warning to stderr. No harm done.
2. If `on-complete.py` commits first, the watcher sees `status = 'done'` (not `running`) and skips the task. It may pick it up for reporting if `reported = 0`.
3. The `total_tasks` counter could be incremented twice in the race case. This is a minor cosmetic issue. A more robust approach would use `UPDATE agents SET total_tasks = (SELECT COUNT(*) FROM tasks WHERE session_id = ? AND status IN ('done','failed','timeout','killed'))` but the simple increment is acceptable for MVP.
