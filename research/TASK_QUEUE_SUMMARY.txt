CLAUDE BRIDGE TASK QUEUE ARCHITECTURE — RESEARCH SUMMARY
========================================================

Date: 2026-03-26
Scope: Design a task queue for Claude Bridge with Telegram input + SQLite storage + multi-session polling
Status: COMPLETE (Ready for implementation)

KEY FINDINGS
============

1. SQLite + Polling is the CLEAR WINNER for MVP
   - No external dependencies (sqlite3 is stdlib)
   - ACID transactions guarantee no duplicates
   - WAL mode provides crash recovery
   - 5-30s latency is acceptable for task queue
   - Scales easily to 1000+ tasks
   - Debugging is trivial (just query SQL)

2. Concurrency safety is SOLVED by SQLite IMMEDIATE transactions
   ```sql
   BEGIN IMMEDIATE;
   UPDATE tasks SET status='running' WHERE id=? AND status='pending';
   -- Only ONE session succeeds, others get rowcount==0
   COMMIT;
   ```
   This prevents duplicate execution 100% reliably.

3. Crash recovery is AUTOMATIC with WAL mode
   - No manual intervention needed
   - Database recovers on next open()
   - Stale tasks detected by cron watcher (run every 1 min)
   - Auto-reset to pending for retry

4. Failure scenarios are WELL-HANDLED
   - Session crash: Cron detects via heartbeat timeout, resets task
   - Task timeout: Configured per-agent (default 600s)
   - Database locked: 5s timeout + exponential backoff
   - Invalid task: DB constraints enforce at write time

DATA FLOW
=========

1. USER CREATES TASK (Telegram)
   /spawn coder-my-app "Fix login bug"
   → TelegramChannel.create_task()
   → INSERT INTO tasks (agent_name='coder-my-app', description='Fix login bug', status='pending')
   → ✅ Task queued

2. SESSION POLLS FOR WORK
   TaskPoller._poll_loop() runs every 5 seconds:
   → BEGIN IMMEDIATE
   → SELECT * FROM tasks WHERE status='pending' AND agent_name='coder-my-app' LIMIT 1
   → UPDATE tasks SET status='running', assigned_to='session_xyz' WHERE id='task_001'
   → COMMIT
   → ✅ Task claimed (atomically)

3. SESSION EXECUTES TASK
   → subprocess.run(['claude', '--project', '~/my-app', '-p', 'Fix login bug'])
   → Waits up to 600 seconds
   → Captures output + return code

4. SESSION REPORTS RESULT
   → UPDATE tasks SET status='completed', output='...', exit_code=0 WHERE id='task_001'
   → Update session heartbeat

5. CRON WATCHER REPORTS BACK
   Every 5 minutes: CompletionReporter.report_completed()
   → SELECT * FROM tasks WHERE status IN ('completed', 'failed') AND reported_at IS NULL
   → FOR each task: send_telegram_message()
   → UPDATE tasks SET reported_at=NOW()

KEY TECHNICAL DECISIONS
=======================

Q: SQLite vs YAML?
A: SQLite. YAML has race conditions, SQLite has ACID + built-in locking.

Q: How prevent duplicate execution?
A: IMMEDIATE transaction mode. Only one session's UPDATE succeeds.

Q: What polling interval?
A: 5 seconds (good balance between latency and CPU)
   Can back off to 30s if queue is empty.

Q: Can Claude Code CLI monitor task queue?
A: Not directly. Use wrapper script with TaskPoller class (runs in each session).

Q: What if session crashes mid-task?
A: Cron watcher checks heartbeat every 1 min.
   Resets stale tasks to 'pending' for retry (up to 3x).

Q: Database locking/contention?
A: SQLite PRAGMA busy_timeout=5000 (wait 5 sec before timeout).
   Multiple sessions can read simultaneously.
   Only writes block each other (briefly).

CONCLUSION
==========

SQLite + Polling is the clear winner for Claude Bridge MVP because:

1. SIMPLICITY: 1 week to implement, no external dependencies
2. RELIABILITY: ACID transactions guarantee correctness
3. SCALABILITY: Handles 1000+ tasks efficiently
4. DEBUGGABILITY: Query the database directly
5. CRASH-SAFE: WAL mode auto-recovery

When you outgrow this (unlikely in MVP), the migration path is clear:
→ Daemon IPC (if you need real-time)
→ Message Queue (if you need distributed systems)

But don't optimize for problems you don't have yet. This architecture is 
"good enough" for years of operation at typical startup scale.
