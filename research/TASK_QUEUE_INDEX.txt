QUESTIONS ANSWERED
==================

Q: SQLite vs YAML for task queue?
A: SQLite. YAML has race conditions, SQLite has ACID + locking.
   Document: TASK_QUEUE_DESIGN.md section 1

Q: How ensure tasks picked up by correct session?
A: SQL WHERE clause filters by agent_name.
   Multiple sessions for same agent race for same tasks.
   Document: TASK_QUEUE_DESIGN.md section 5

Q: What's the polling interval?
A: 5 seconds (good balance). Can back off to 30s if queue empty.
   Document: TASK_QUEUE_DECISIONS.md decision #3

Q: How prevent duplicate task execution?
A: BEGIN IMMEDIATE transaction + rowcount check.
   Only one session's UPDATE succeeds.
   Document: TASK_QUEUE_DESIGN.md section 6

Q: Can Claude Code CLI monitor task queue?
A: Not directly. Use TaskPoller class in background thread.
   Document: TASK_QUEUE_DESIGN.md section 7

Q: Pros/cons: simplicity vs responsiveness vs reliability?
A: Polling is simple+reliable. Daemon is more responsive.
   Trade-offs documented with decision matrix.
   Document: TASK_QUEUE_COMPARISON.md

DATABASE SCHEMA SUMMARY
=======================

tasks:
  id, created_at, created_by
  description, project_path, agent_name, context
  status (pending|running|completed|failed|cancelled)
  assigned_to, assigned_at, started_at, completed_at, reported_at
  output, files_changed, error_message, exit_code
  priority, retry_count, max_retries

sessions:
  id, agent_name, state, last_heartbeat, created_at, current_task_id

agents:
  name (PK), project_path, role, created_at, last_used

audit_log:
  id, timestamp, event, task_id, session_id, details

KEY TECHNICAL DECISIONS
=======================

✅ Storage: SQLite with WAL mode
✅ Concurrency: Polling + atomic claiming (BEGIN IMMEDIATE)
✅ Poll interval: 5 seconds (with 30s backoff)
✅ Task timeout: 600 seconds (10 minutes)
✅ Max retries: 3
✅ Stale timeout: 300 seconds (5 minutes)
✅ Heartbeat: Every poll cycle
✅ Recovery watcher: Every 60 seconds
✅ Completion reporter: Every 300 seconds
✅ Configuration: Single YAML file (~/.claude-bridge/config.yaml)
✅ Debugging: Direct SQL queries on database

MIGRATION PATH (IF NEEDED)
==========================

Phase 1 (Now):     SQLite Polling (simple, reliable)
Phase 2 (Months):  Daemon IPC (if real-time needed)
Phase 3 (Year+):   Message Queue (if distributed scale)

But Phase 1 is "good enough" for years of typical startup operation.
Don't optimize for future problems you don't have yet.

WHEN TO USE THIS APPROACH
=========================

✅ Building Claude Bridge MVP
✅ <100 tasks per day
✅ 5-30 second latency acceptable
✅ Single machine deployment
✅ Want simple, reliable operation
✅ Easy debugging priority

❌ Not for:
  - Real-time dispatch required (<1 second)
  - 1000+ tasks per day (move to message queue)
  - Distributed systems (move to daemon/message queue)
  - Complex task dependencies
