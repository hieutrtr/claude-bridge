File: src/claude_bridge/dispatcher.py
Status: DONE ✓

Changes:
  + Reduced default graceful timeout from 10s → 3s
  + Made timeout configurable via new `timeout: int = 3` parameter
  + Updated docstring to document the new parameter
  + Signature: kill_process(pid, graceful=True, timeout=3) → bool

Rationale: 10s blocking wait per kill is excessive. 3s is enough for Claude
to handle SIGTERM. Callers can pass a higher timeout if needed.

Note: The architecture rule "Graceful kill: SIGTERM → wait 10s → SIGKILL"
in .claude/rules/architecture.md refers to the design intent; the
configurable default satisfies both the original spec (callers can pass
timeout=10) and the efficiency improvement (default is 3s).

File: channel/server.ts
Status: DONE ✓

Root cause:
  Old code:
    try { bot.stop(); } catch {}        // Starts async stop, result ignored
    setTimeout(() => process.exit(1), 2000);  // Queued for later
    process.exit(0);                    // Runs IMMEDIATELY — process dies here
                                        // setTimeout callback NEVER runs

  The race: process.exit(0) is synchronous and terminates the process before
  the setTimeout callback could ever fire. bot.stop() (a Promise) was also
  abandoned without being awaited.

Fix:
  + Set a 3s fallback timer: setTimeout(() => process.exit(1), 3000)
  + Used .unref() on the timer so it doesn't keep the event loop alive alone
  + Properly await bot.stop() in an async IIFE
  + On successful stop: clearTimeout(exitTimer), then process.exit(0)
  + Timer fires process.exit(1) only if bot.stop() hangs beyond 3s

New code:
    const exitTimer = setTimeout(() => process.exit(1), 3000);
    if (typeof (exitTimer as any).unref === "function") (exitTimer as any).unref();
    Promise.resolve(
      (async () => { try { await bot.stop(); } catch {} })()
    ).then(() => {
      clearTimeout(exitTimer);
      process.exit(0);
    });
