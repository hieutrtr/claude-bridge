     skips if already present — idempotent)
  2. Prints ⚠️ WARNING about plaintext token and not committing to git

Why token stays in .mcp.json:
  - The channel server (bun) reads TELEGRAM_BOT_TOKEN from env
  - MCP config's `env` section is the only supported mechanism to pass env vars
    to the server process from Claude Code's .mcp.json
  - Using "${TELEGRAM_BOT_TOKEN}" as a placeholder would cause the bun process
    to receive literal string "${TELEGRAM_BOT_TOKEN}", not the actual token →
    authentication would fail
  - Mitigation: gitignore entry prevents accidental git commit of the token

Output example:
  .mcp.json → /path/to/botdir/.mcp.json
  .gitignore ← added .mcp.json entry
  ⚠️  WARNING: .mcp.json contains your Telegram bot token in plaintext.
     It has been added to .gitignore. DO NOT commit .mcp.json to git.

New method in BridgeDB (db.py):
  def atomic_check_and_create_task(
      self, session_id, prompt, channel, channel_chat_id, channel_message_id
  ) -> tuple[int | None, bool]:

  Mechanism:
    1. Saves isolation_level, switches conn to autocommit (isolation_level=None)
    2. Executes BEGIN EXCLUSIVE → takes exclusive write lock on the DB
    3. Checks for running task (status='running')
    4. If running: COMMIT, return (None, True) — caller should queue
    5. If free: INSERTs task with status='running' (not 'pending'), COMMIT,
       returns (task_id, False) — caller should spawn
    6. On exception: ROLLBACK, re-raise
    7. Restores isolation_level in finally block

  Why status='running' in the INSERT:
    - Inserting as 'running' immediately prevents the second concurrent dispatch
      from seeing "no running task" and also spawning
    - If spawn fails, the task is already in 'running' state and will be
      detected as a dead PID by the watcher (which marks it 'failed')
