# Context Generation Specification (CLAUDE.md)

## Overview

Bridge generates **CLAUDE.md files** from profiles. These files contain actual instructions that Claude Code agents read.

**Key principle:** CLAUDE.md is **context for the agent**, not configuration. It's prose, not data.

---

## 1. Multi-Layer CLAUDE.md

### 1.1 Generation Strategy

Bridge generates CLAUDE.md at **multiple layers**:

```
{project}/
├── CLAUDE.md                        ← Project-wide rules, stack, general context
├── src/
│   ├── auth/
│   │   ├── CLAUDE.md                ← Auth-specific rules and constraints
│   │   └── session.ts
│   └── payments/
│       ├── CLAUDE.md                ← Payment-specific rules (extra strict)
│       └── webhook.ts
└── tests/
    └── CLAUDE.md                    ← Test-specific instructions
```

**Why multiple layers?**
- Project-level: common for all tasks
- Module-level: specific rules for that area
- When agent works in `src/auth/`, it reads both `project/CLAUDE.md` + `src/auth/CLAUDE.md`
- Allows **different constraints per area** (payments more strict than auth)

### 1.2 Which Layers to Generate

**Always generate:**
- `{project}/CLAUDE.md` — project root, mandatory

**Conditionally generate:**
- For each `sensitive: true` in profile.key_files → generate layer CLAUDE.md
- Example: if `src/payments/**` marked sensitive → generate `src/payments/CLAUDE.md`

**Optional generate:**
- User can mark specific dirs in profile as "should have own CLAUDE.md"

## 2. Template System

### 2.1 Base Templates

**Location:** `claude_bridge/templates/claude_md_base.jinja2`

```jinja2
# Agent: {{ agent_name }}
<!-- Generated by Claude Bridge v{{ bridge_version }} -->
<!-- Last updated: {{ timestamp }} -->
<!-- Do not edit manually (changes will be overwritten) -->

## 🎭 Role
You are a **{{ display_name }}** working on {{ project_path }}.

### Technology Stack
{{ stack_description }}

### Project Overview
{{ project_description }}

---

## 📁 Project Structure
{{ structure_description }}

---

## 📎 Critical Files to Know
{% for file in critical_files %}
- **{{ file.path }}** — {{ file.reason }}
{% endfor %}

---

## 🔒 Rules — NEVER BREAK THESE
{% for rule in hard_rules %}
{{ loop.index }}. {{ rule.text }}
{% endfor %}

---

## 📐 Conventions & Patterns
### Code Style
{% for convention in code_conventions %}
- {{ convention }}
{% endfor %}

### Known Patterns
{% for pattern in learned_patterns %}
- {{ pattern }}
{% endfor %}

---

## 📣 How to Report Results
{% for reporting_rule in reporting_rules %}
- {{ reporting_rule }}
{% endfor %}

---

## 🧠 Context — Lessons From Past Work
{% for issue in known_issues %}
- **{{ issue.description }}**
  Workaround: {{ issue.workaround }}
{% endfor %}

---

## 📋 If Session Gets Compacted
Read `.bridge/dev/context.md` to resume properly.
```

### 2.2 Layer-Specific Template

For sensitive dirs, add extra section:

```jinja2
# ⚠️ SENSITIVE AREA: {{ layer_name }}

This directory handles {{ sensitivity_reason }}.

## Extra Rules for This Area
{% for rule in layer_specific_rules %}
- {{ rule }}
{% endfor %}

## What Requires Confirmation
{% for item in requires_confirmation %}
- {{ item }}
{% endfor %}

---
{{ base_template }}
```

### 2.3 Template Variables

```python
@dataclass
class TemplateVariables:
    # Basic
    agent_name: str                   # e.g., "coder-my-app"
    display_name: str                 # e.g., "Senior Full-stack Developer"
    project_path: str                 # ~/projects/my-app
    bridge_version: str               # e.g., "1.0.0"
    timestamp: str                    # ISO8601 datetime

    # Project info
    project_description: str          # From README or profile
    stack_description: str            # "Next.js 15, TypeScript, Prisma, React Query"
    structure_description: str        # Overview of dirs

    # From profile
    critical_files: list[CriticalFile]  # Sorted by importance
    hard_rules: list[str]
    code_conventions: list[str]
    learned_patterns: list[str]
    known_issues: list[KnownIssue]
    reporting_rules: list[str]

    # For layer-specific
    layer_name: str                   # e.g., "Payments Module"
    sensitivity_reason: str           # e.g., "payment processing"
    layer_specific_rules: list[str]
    requires_confirmation: list[str]
```

- **Profile updated** — Any field changed → regenerate
- **Enhancement applied** — New rules added → regenerate

- **Manual regeneration** — `/regenerate-context` command

## 6. Special Cases

### 6.1 Large Projects

For very large projects with many sensitive areas:
- Generate project-level CLAUDE.md (always)
- Generate only N most important layer-specific CLAUDE.md files
- Log which layers were skipped
- Suggest adding more layers if needed

### 6.2 Dynamic Content

Some variables are computed dynamically:
- `{{ git_status }}` — Current branch, uncommitted changes
- `{{ recent_changes }}` — Files changed in last N commits
- `{{ test_status }}` — Are tests passing?

These can be:
- **Static** (computed once, cached)
- **Dynamic** (computed on agent startup via hook)

## 7. Markdown Quality

### 7.1 Standards

All generated CLAUDE.md must:
- ✓ Valid Markdown syntax
- ✓ Headings use proper hierarchy (##, ###, not ####)
- ✓ Code blocks use language tags
- ✓ Lists use consistent formatting
- ✓ Links are relative or absolute (not ambiguous)
- ✓ No broken references
- ✓ Readable in plaintext (no fancy Unicode)

## 8. Error Handling

| Error | Recovery |
|---|---|
| Template not found | Log error, use fallback template |
| Variable missing | Use default value, log warning |
| File write fails | Keep existing file, log error |
| Markdown invalid | Log warning, don't write |
| Profile changed during generation | Restart generation |
