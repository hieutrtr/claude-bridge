## What This Task Does
Design and implement a channel abstraction that lets Bridge Bot work with Telegram, Discord, and Slack through a unified interface. Add `channel` column to tasks table.

## Key Architecture Constraints
- Bridge Bot is the single Claude Code session that talks to all channels via MCP
- Each channel is an MCP server plugin — no direct API calls from bridge code
- Channel abstraction is data-only: stores channel source + chat_id per task
- Formatting is channel-specific (Telegram MarkdownV2, Slack mrkdwn, Discord markdown)
- One Bridge Bot handles all channels (not one per channel)

## Design

### Channel Data Model
```sql
-- Add to tasks table
ALTER TABLE tasks ADD COLUMN channel TEXT DEFAULT 'cli';
ALTER TABLE tasks ADD COLUMN channel_chat_id TEXT;
ALTER TABLE tasks ADD COLUMN channel_message_id TEXT;
```

Channel values: 'cli', 'telegram', 'discord', 'slack'

### Channel Module (channel.py)
```python
# Format a message for a specific channel
def format_message(channel: str, text: str) -> str

# Parse channel info from dispatch context
def parse_channel_context(channel: str, chat_id: str, message_id: str | None) -> dict
```

### Formatter Differences
- **Telegram**: MarkdownV2 (escape special chars)
- **Discord**: Standard markdown
- **Slack**: mrkdwn (different from markdown — `*bold*`, `_italic_`, `~strike~`)
- **CLI**: Plain text

## Test Plan
Happy path:
- format_message for each channel type
- Channel context stored in task row
- Task history includes channel info

Edge cases:
- Unknown channel falls back to plain text
- Empty chat_id handled
- Special chars escaped per channel

Errors:
- Invalid channel type
