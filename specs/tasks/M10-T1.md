## What This Task Does
Research how Claude Code hooks work for permission interception, design the relay mechanism, implement it.

## Key Architecture Constraints
- Claude Code hooks fire on lifecycle events (PreToolUse, Stop, etc.)
- Hook types: command, prompt, agent, http
- Hook can block execution (exit code 2 = deny)
- Agent .md frontmatter can define hooks per agent
- Permission relay: hook blocks → writes request to file → polls for response → returns exit code

## Design: File-Based Permission Relay

```
Agent runs dangerous command (e.g., git push)
  → PreToolUse hook fires
  → permission-relay.py writes request to pending-permissions/{request_id}.json
  → permission-relay.py polls for response file (timeout 5 min)
  → Bridge Bot / watcher detects pending permission
  → Sends Telegram message (or prints for now)
  → User approves/denies
  → Response written to pending-permissions/{request_id}.response.json
  → permission-relay.py reads response, returns exit code (0=allow, 2=deny)
```

## Implementation Plan
1. permission-relay.py — hook script that blocks and polls
2. DB: permissions table (request_id, session_id, tool, command, status, response)
3. CLI: pending-permissions command to list/approve/deny
4. Agent .md: add PreToolUse hook for dangerous commands
5. Tests with mocked hook invocation

## Test Plan
Happy path:
- Permission request created in DB
- Approve command updates status, returns allow
- Deny command updates status, returns deny

Edge cases:
- Timeout auto-denies
- Multiple pending permissions
- Permission for non-existent request

Errors:
- Hook script handles DB errors gracefully
